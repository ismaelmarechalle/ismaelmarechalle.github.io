SQL Inventory Query Library

A normalized SQLite supply chain database with 12 annotated business queries covering reorder points, stockout frequency, supplier performance, and more.

🗄️ Deliverable: SQLite DB + queries.sql + README
📋 Tables: 5 normalized tables
🔍 Queries: 12 annotated business queries

The Problem

Supply chain analysts increasingly need SQL to query warehouse management systems, ERP databases, and data warehouses directly. Most junior candidates claim "SQL knowledge" but have no artifacts to prove it. This project creates a realistic, queryable database and demonstrates SQL literacy through 12 queries that answer real business questions.

Database Schema

The database is generated by a Python script using SQLite (no server required). Five normalized tables:

-- Schema overview CREATE TABLE products ( product_id INTEGER PRIMARY KEY, sku TEXT, name TEXT, category TEXT, unit_price REAL, reorder_point INTEGER, lead_time_days INTEGER ); CREATE TABLE suppliers ( supplier_id INTEGER PRIMARY KEY, name TEXT, country TEXT, rating REAL -- 1.0 to 5.0 ); CREATE TABLE stock_movements ( movement_id INTEGER PRIMARY KEY, product_id INTEGER REFERENCES products, movement_date DATE, quantity INTEGER, movement_type TEXT -- 'in' | 'out' | 'adjustment' ); CREATE TABLE purchase_orders ( po_id INTEGER PRIMARY KEY, supplier_id INTEGER REFERENCES suppliers, product_id INTEGER REFERENCES products, order_date DATE, promised_date DATE, received_date DATE, quantity_ordered INTEGER, unit_cost REAL ); CREATE TABLE warehouses ( warehouse_id INTEGER PRIMARY KEY, name TEXT, location TEXT, capacity_m2 INTEGER );

Query Library — 12 Business Queries

Q1 — SKUs below reorder point

Flags products where current stock has fallen below the defined reorder point. Core replenishment alert.

SELECT p.sku, p.name, p.reorder_point, SUM(CASE WHEN m.movement_type = 'in' THEN m.quantity WHEN m.movement_type = 'out' THEN -m.quantity ELSE 0 END) AS current_stock FROM products p JOIN stock_movements m ON p.product_id = m.product_id GROUP BY p.product_id HAVING current_stock < p.reorder_point;

Q4 — Average lead time per supplier (last 90 days)

Measures actual vs. promised lead time to identify suppliers who are systematically slow.

SELECT s.name AS supplier, ROUND(AVG(julianday(po.received_date) - julianday(po.order_date)), 1) AS avg_actual_lead_time, ROUND(AVG(julianday(po.promised_date) - julianday(po.order_date)), 1) AS avg_promised_lead_time, COUNT(*) AS po_count FROM purchase_orders po JOIN suppliers s ON po.supplier_id = s.supplier_id WHERE po.order_date >= date('now', '-90 days') GROUP BY s.supplier_id ORDER BY avg_actual_lead_time DESC;

All 12 Queries

  • Q9 — Suppliers providing >80% of total purchase value (spend concentration)
  • Q10 — Average days between reorder trigger and receipt
  • Q11 — SKUs with highest demand variability (standard deviation of weekly outflows)
  • Q12 — Purchase order fill rate by supplier (ordered qty vs. received qty)

All queries are annotated: each one includes a comment block explaining the business question it answers, the key join logic, and one interpretation note — showing that SQL is being used to think, not just retrieve.
